ملخص سريع

الباك إند عبارة عن مجموعة واجهات API مسؤولة عن تسجيل المستخدمين، تسجيل الدخول، إدارة التبرعات، الرسائل، الإشعارات، والمطابقة. كل طلب يمر على طبقة الأمان (JWT) ثم يتعامل مع قاعدة البيانات.

التقنيات المستخدمة

ASP.NET Core Web API
Entity Framework Core
SQL Server
JWT Authentication
BCrypt لتشفير كلمات المرور
Swagger لتوثيق وتجربة الواجهات

هيكل المشروع

yad-elawn/ ├─ frontend/ ├─ backend/ │ └─ api/YadElAwn.Api/ ├─ ai/ ├─ database/ ├─ docs/

مسار العمل للمستخدم الجديد (User Journey)

1) المستخدم يختار نوعه (متبرع/جمعية/مستفيد/أدمن).
2) يرسل طلب تسجيل إلى:
POST /api/registrations/{role}
النظام يحفظ المستخدم في جدول المستخدمين ويضيفه في جدول الدور المناسب.
3) بعد التسجيل، المستخدم يعمل تسجيل دخول:
POST /api/auth/login
الباك إند يتحقق من الإيميل والباسورد (المشفر) ويرجع Token.
4) أي طلب محمي لازم يتبعت مع الهيدر:
Authorization: Bearer <TOKEN>

مسار التبرع (Donation Flow)

1) المتبرع يعمل Login ويحصل على Token.
2) ينشئ تبرع جديد:
POST /api/donations
3) الباك إند ينشئ سجل في جدول Donation ويضيف تفاصيل النوع (Food/Clothes/Medicine/Medical_Supplies).
4) يمكن تحديث الحالة:
PATCH /api/donations/{id}/status
5) عرض التبرعات المتاحة:
GET /api/donations/available

مسار الرسائل والإشعارات

الرسائل:
POST /api/messages
GET /api/messages/inbox/{userId}
الإشعارات:
POST /api/notifications
GET /api/notifications/{userId}

مسار المطابقة (Matches)

إنشاء مطابقة بين تبرع وجمعية ومستفيد:
POST /api/matches
عرض المطابقات:
GET /api/matches

ما الذي يحدث داخل الباك إند (تقنيًا)

  • كل طلب يدخل Controller مناسب.
  • التحقق من البيانات يتم عبر DTOs.
  • يتم الحفظ في الداتابيز عبر EF Core.
  • الطلبات المحمية تمر على JWT Middleware.
  • أي استجابة يتم تحويلها لـ JSON بشكل آمن.

التحديات والحلول

  • Swagger لا يظهر: تم ضبط إعداداته ليظهر على المسار الرئيسي.
  • خطأ 500 في التبرع: حل دورة JSON باستخدام IgnoreCycles.
  • اختلاف أسماء الجداول: تم تثبيت أسماء الجداول في EF Core.

سؤال وجواب (احتمالات الأسئلة)

س: إزاي الفرونت يستخدم الـ API؟
ج: Register → Login → يستخدم التوكن في الهيدر لأي طلب محمي.
س: ليه استخدمنا JWT؟
ج: عشان الأمان ولتأكيد هوية المستخدم.
س: الباسورد بيتخزن إزاي؟
ج: مشفر بـ BCrypt.
س: ليه فصلنا الجداول حسب الدور؟
ج: عشان كل دور له خصائص مختلفة وتسهيل التحكم.
س: إزاي نعرض التبرعات المتاحة؟
ج: باستخدام GET /api/donations/available.
س: لو واجهنا 500 Error؟
ج: نراجع اللوج ونحدد هل المشكلة في الداتابيز أو في الموديل أو في الـ JSON.
س: هل النظام جاهز للتسليم؟
ج: نعم، تم اختبار التسجيل والدخول والتبرعات والرسائل والإشعارات والمطابقات.
